# Method: Trace-Selection Pipeline / 方法：Trace 选择风险实验管线

This note records the current experimental method in paper-style language. / 本文档用论文 Method 的写法记录当前实验方法。

## 1. Scientific Object / 科学对象

We study `answer-correct but process-invalid` (`ACPI`) traces: the final answer is correct, but at least one visible reasoning step is mathematically, semantically, logically, or procedurally invalid. / 我们研究 `答案正确但过程无效` 的 trace：最终答案对，但可见推理中至少一步在数学、语义、逻辑或程序执行上是错的。

We separate three labels. / 我们区分三种标签：

- `strict process validity / 严格过程有效性`: every visible step must be valid. If a wrong step appears anywhere, the trace is invalid even if later repaired. / 只要可见过程出现错步，整段 trace 在 strict 口径下无效。
- `repair-aware validity / 修复感知有效性`: a trace can be accepted if it explicitly discards the wrong step and later gives a self-consistent correct proof. / 如果后文明确丢弃错步并给出自洽正确证明，可按 repair-aware 口径接受。
- `unrepaired ACPI / 未修复 ACPI`: the final answer is correct, but the final retained proof still depends on a wrong key step. / 最终答案正确，但最终保留的证明仍依赖错误关键步骤。

This distinction is central: many natural hard-task traces are not clean proofs; they are more like scratchpads with repair. / 这一区分很重要：很多自然困难题 trace 不是干净证明，而更像带修复的草稿。

## 2. Trace Sources / Trace 来源

### 2.1 Controlled Synthetic Traces / 受控合成 Trace

Controlled traces are built from small task templates where we know the intended operation and can construct both valid and invalid processes. / 受控 trace 来自小型任务模板，我们知道正确操作，因此能构造正确过程和错误过程。

For each task family, we create: / 每个任务族通常包含：

- a problem statement / 题目；
- a valid trace / 正确过程；
- one or more invalid traces with the same final answer or an answer-coherent ending / 最终答案相同或后文答案自洽的错误过程；
- offline metadata: gold answer, process-valid label, error span, repair flag, language route. / 离线元数据：答案、过程标签、错误 span、是否修复、语言路线。

The verifier never receives the offline metadata unless an experiment is explicitly labeled answer-anchor or non-prevalence. / 除非实验明确标注为 answer-anchor 或非发生率实验，verifier 不会看到这些离线元数据。

Controlled task families already used include: / 已使用的受控任务族包括：

- percentage base / 百分比基准；
- unit and scale conversion / 单位与数量级转换；
- multilingual quantifiers and inequalities / 多语言量词与不等式；
- counting order and table interpretation / 计数顺序与表格读取；
- code execution trace / 代码执行 trace；
- proof validity and algebraic transformation / 证明有效性与代数变换；
- geometry notation / 几何符号；
- suspicious-but-valid controls / 可疑但正确控制组。

`E132` extends this with five variants: / `E132` 扩展了五类变体：

- `clean_valid`: correct and ordinary. / 干净正确。
- `suspicious_valid_marker`: correct but includes words such as wait, check, verify. / 有“等一下/检查/核对”等词但过程正确。
- `suspicious_valid_alternative`: correct with a valid alternate route. / 使用替代正确路线。
- `low_conf_valid`: correct but worded cautiously. / 语气不确定但过程正确。
- `repaired_strict_invalid`: contains a wrong step and later repairs it. / 先错后修，strict 无效。

### 2.2 Natural Hard-Task Traces / 自然困难题 Trace

Natural traces are generated by P0 models on hard math/AIME-style tasks under no-gold prompts. / 自然 trace 来自 P0 模型在困难数学/AIME-style 任务上的无答案生成。

The generation prompt may be neutral, answer-first, or self-check. / 生成 prompt 包括中性、先给答案、主动自检等形式。

Only final-correct rows are sent to process audit. The audit labels whether the visible process is strict-valid, repaired ACPI, unrepaired ACPI, or a truncation/fallback boundary. / 只有最终答案正确的行进入过程审计；审计标注 strict-valid、repaired ACPI、unrepaired ACPI 或截断边界。

The current natural evidence shows many repaired strict ACPI rows and a small number of unrepaired ACPI rows. / 当前自然证据显示 repaired strict ACPI 较多，未修复 ACPI 低频但存在。

## 3. Verifier Objectives / Verifier 目标函数

We use deterministic option-logprob scoring rather than free-form generation for most verifier experiments. / 大多数 verifier 实验使用确定性的选项 logprob 打分，而不是开放生成。

The verifier receives `problem + visible trace` and we score the log-probability margin between labels such as `Yes` and `No`, or between pairwise options such as `A` and `B`. / verifier 输入是 `题目 + 可见 trace`，输出用 `Yes/No` 或 `A/B` 的 token logprob margin 打分。

Main objective types: / 主要目标类型：

- `plain pointwise Yes/No / 普通单点 Yes/No`: judge whether the trace is valid. This can over-accept answer-correct invalid traces. / 判断 trace 是否有效，容易被正确答案锚定。
- `strict pointwise Yes/No / 严格单点 Yes/No`: any visible wrong step means No. / 任何可见错步都判 No。
- `answer-blind / 去答案锚定`: remove or mask final answer to test answer anchoring. / 移除或遮蔽最终答案，测试答案锚定。
- `sibling comparison / 同胞对比`: present a valid and invalid sibling trace for the same problem and ask which process is better. / 同题给一对正确/错误 sibling trace，让模型比较过程。
- `label-free two-pass / 无标签双遍`: score each trace separately with a label-free invalidity question and compare scores afterward. / 分别评价两条 trace，再离线比较，不强迫 A/B 标签。

## 4. Hidden-State and Component Signals / 隐藏层与组件信号

We train process-validity directions from controlled valid/invalid rows, then project hidden activations of new traces onto these directions. / 我们用受控正确/错误行训练过程有效性方向，再把新 trace 的隐藏激活投影到这些方向上。

Components inspected include: / 检查的组件包括：

- residual hidden state / residual 隐藏状态；
- MLP output or post-feedforward state / MLP 输出或前馈后状态；
- token-mixer or attention-related output / token-mixer 或注意力相关输出；
- norm outputs when available / 可用时的 norm 输出。

The signal is interpreted as `process-risk evidence`, not a pure error detector. / 该信号解释为“过程风险证据”，不是纯错误检测器。

Prefix replay is used to locate where the signal changes. / 我们用前缀重放定位信号变化位置：

- before an error / 错误前；
- at error-span end / 错误 span 结束；
- at repair marker / 修复标记；
- after repair / 修复后；
- at completion / 完成态。

## 5. E136 Adaptive Checking / E136 自适应检查

E136 is a stage-1 post-hoc policy simulation, not online generation-time intervention. / E136 是第一阶段后验策略模拟，不是在线生成时激活干预。

Pipeline / 流程：

1. Run the normal pointwise verifier on a complete trace. / 先对完整 trace 运行普通单点 verifier。
2. Replay selected prefixes and compute hidden process-risk scores. / 重放若干前缀并计算 hidden 过程风险分数。
3. If a non-marker-only prefix has a negative process-validity score, mark the trace as `hidden-triggered`. / 如果非 marker-only 前缀的过程有效性分数为负，则触发 hidden 检查。
4. If no trigger fires, keep the base verifier decision. / 未触发则保留基线 verifier 决策。
5. If trigger fires, run a second verifier. / 触发则调用二次 verifier。

The second verifier has two forms. / 二次 verifier 有两种形式：

- `global check / 全局复查`: it receives the full problem and full visible trace. / 输入完整题目和完整 trace。
- `local check / 局部复查`: it receives the full problem, full visible trace, and a visible excerpt around the hidden-triggered prefix. / 输入完整题目、完整 trace，以及 hidden 触发点附近的可见片段。

The local excerpt is selected by hidden signal only; the prompt does not include gold answer, manual label, or error-span annotation. / 局部片段只由 hidden 信号选择；prompt 不包含 gold、人工标签或 error-span 标注。

Current E136 finding / 当前 E136 发现：

- Qwen and Gemma31 support hidden-triggered low-cost checking. / Qwen 和 Gemma31 支持 hidden 触发的低成本复查。
- Gemma26-A4B exposes a boundary: hidden trigger catches risk, but local check may still accept invalid traces if the objective/readout reads the trace as a repairable draft. / Gemma26-A4B 暴露边界：hidden 能抓到风险，但二次检查如果把 trace 当可修复草稿读，仍可能放过错步。

## 7. Method Boundaries / 方法边界

- Current hidden signals are observational unless an experiment explicitly performs activation patching or steering. / 当前 hidden 信号若没有 patch/steering，就属于观察性证据。
- The second verifier is a filter/recheck layer, not proof that the base model would self-correct without prompting. / 二次 verifier 是筛选/复查层，不等于证明原模型会无提示自我纠错。
- Controlled ACPI shows a robust failure mode; natural unrepaired ACPI currently remains low-frequency. / 受控 ACPI 证明风险模式稳健；自然未修复 ACPI 目前仍低频。
- All claims must be separated by mode: `NG/non-thinking`, `TG/thinking`, and deterministic direct-verifier replay. / 所有结论必须按 non-thinking、thinking 和确定性 verifier replay 分开写。
